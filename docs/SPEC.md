## 1. Идея продукта

B2B-платформа для заказа товаров (мебель и др.) у заводов **через консультантов**.
Главное правило: **клиент никогда не общается с заводом напрямую**. Цепочка:

```
КЛИЕНТ  ⇄  КОНСУЛЬТАНТ  ⇄  ЗАВОД (factory_admin)
        (чат 1)        (чат 2, скрыт от клиента)
```

- Клиент выбирает завод/товар → пишет консультанту этого завода → оформляет заявку.
- Консультант при необходимости согласует с заводом (отдельный скрытый чат) наличие/срок/цену — **для кастома обязательно**, для каталога по необходимости.
- Консультант присылает клиенту итоговое предложение (цена, кол-во, срок).
- Клиент соглашается → завод начинает производство.

**Отложено на потом:** трекинг доставки и монетизация (комиссия платформы).
Цена в заказе есть, но платформа с неё пока ничего не берёт.

## 2. Роли (4)

| Роль | Кто это | Кратко |
|------|---------|--------|
| `platform_admin` | Администратор платформы | Суперпользователь: создаёт заводы и их админов, создаёт консультантов и клиентов, управляет категориями, модерирует отзывы, видит всё |
| `factory_admin` | Владелец/админ **одного** завода | Управляет своим заводом: каталог товаров, консультанты завода, статистика, отвечает консультантам в чате согласования |
| `consultant` | Сотрудник **одного** завода | Принимает заявки клиентов, чат с клиентом, чат с заводом, формирует предложение; есть рейтинг/отзывы |
| `client` | Покупатель | Регистрируется сам; смотрит заводы/товары/категории, пишет консультанту, оформляет заявку, подтверждает предложение, оставляет отзыв |

## 3. Создание аккаунтов

| Аккаунт | Кто создаёт |
|---------|-------------|
| `platform_admin` | Сидер / другой платформ-админ |
| `factory_admin` | **Только платформ-админ** — создаёт завод и одновременно его админа (отдельный аккаунт с email+паролем) |
| `consultant` | **Платформ-админ** (к любому заводу) **или** `factory_admin` (к своему заводу) |
| `client` | **Сам регистрируется** (публичная форма) **или** платформ-админ создаёт |

- Саморегистрация доступна **только для роли client**.
- При создании `factory_admin`/`consultant`/`client` админом (или `consultant` — фабрик-админом) создатель **задаёт пароль вручную** (форма: email + имя + пароль). Пользователь может сменить пароль после входа.

## 4. Сущности и поля (модель данных)

### User
`id, email (уникальный), passwordHash, role, name, phone?, avatarUrl?, status (active|blocked), createdAt, updatedAt`
- `factory_admin` владеет заводом через `Factory.ownerUserId`.
- `consultant` привязан к заводу через `ConsultantProfile.factoryId`.

### ConsultantProfile (профиль консультанта)
`id, userId (1:1 с User), factoryId (обязателен — консультант принадлежит заводу),
title?, bio?, languages[]?, online, verified, rating (0..5), reviewsCount, createdAt`
- Консультант **всегда** принадлежит ровно одному заводу.

### Factory (завод)
`id, name, nameCn?, ownerUserId (factory_admin), about?, city?, province?/country?,
photoUrls[]?, establishedYear?, verified, rating (0..5), reviewsCount, createdAt`
- Категории завода **выводятся** из категорий его товаров (отдельной таблицы тегов нет).
- `rating`/`reviewsCount` — агрегат отзывов о заводе (см. Review).

### Category (категория товаров, глобальная)
`id, name, slug (уникальный), icon?, blurb?`
- «Категория → заводы» = заводы, у которых есть активный товар в этой категории.

### Product (каталог завода, производство под заказ)
`id, factoryId, categoryId, name, description?, photoUrls[]?, unit (шт/компл/кг…),
priceLo, priceHi (ориентир-диапазон за единицу), moq (мин. партия),
leadTime (срок производства, напр. «30–45 дней»), active (bool), createdAt`
- **Make-to-order:** склада/остатка нет. Это то, что завод **может произвести**.
- Управляет `factory_admin` своего завода (CRUD).
- Клиент видит: название, фото, описание, **ориентир-диапазон цены**, MOQ, срок, категорию.
- Консультант видит то же + использует как основу для предложения.

### Order (заявка → заказ; единая сущность со статусами)
Создаётся клиентом, проходит весь жизненный цикл.

| Поле | Кто заполняет | Описание |
|------|---------------|----------|
| `id` | система | технический UUID |
| `code` | система | человекочитаемый номер, напр. `ORD-0001` |
| `clientId` | система | автор заявки |
| `factoryId` | система | завод (из выбранного консультанта/товара) |
| `consultantId` | клиент | выбранный консультант завода |
| `productId?` | клиент | если заявка из каталога (иначе null — кастом) |
| `productName` | клиент | название позиции (из товара или своими словами) |
| `requestedQty` | клиент | желаемое кол-во |
| `unit?` | клиент/товар | единица |
| `requirements?` | клиент | пожелания/требования (текст) |
| `desiredDeadline?` | клиент | желаемый срок |
| `status` | система | см. §5 |
| `factoryApproved` | factory_admin | для кастома — подтверждение завода перед предложением (по умолч. false) |
| `quotedUnitPrice?` | консультант | финальная цена за единицу (на этапе proposed) |
| `quotedQty?` | консультант | согласованное кол-во |
| `quotedTotal?` | консультант | итог (qty × unitPrice) |
| `quotedLeadTime?` | консультант | согласованный срок |
| `proposalNote?` | консультант | комментарий к предложению |
| `createdAt, updatedAt` | система | |

### OrderStatusHistory
`id, orderId, status, note?, actorId, createdAt` — журнал смены статусов.

### Thread / Message (чаты)
Два типа тредов, **оба привязаны к конкретному заказу**:

| `kind` | Между кем | Виден кому |
|--------|-----------|-----------|
| `client` | клиент ↔ консультант | клиенту, консультанту, платформ-админу |
| `factory` | консультант ↔ factory_admin | консультанту, factory_admin, платформ-админу. **Клиент НЕ видит** |

`Thread: id, orderId, kind, clientId?, consultantId?, factoryId?, createdAt`
`Message: id, threadId, senderId, body, attachmentUrl?, readAt?, createdAt`
- На один заказ допускается максимум **один** `client`-тред и максимум **один** `factory`-тред
  (`unique(orderId, kind)`). Если тред уже есть, backend возвращает существующий.

### Review (отзыв; две цели — консультант и завод)
После `completed` клиент оценивает по своему заказу **две вещи**:
- **консультанта** (качество сопровождения) → влияет на рейтинг консультанта;
- **завод/товар** → влияет на рейтинг завода.

`Review { id, orderId, authorId (клиент), targetType (consultant|factory),
consultantId?, factoryId?, rating (1..5), text, status (pending|approved|removed), createdAt }`
- До **2 отзывов на заказ**: один консультанту + один заводу (каждого типа — максимум один).
- Публикуется после модерации платформ-админом; затем пересчитывается рейтинг цели
  (`ConsultantProfile.rating` или `Factory.rating`).
- В UI клиента после `completed` — одна форма с двумя блоками (звёзды консультанту + звёзды заводу + тексты).
  Клиент может отправить оба отзыва сразу или только один из блоков; второй можно добавить позже.

## 5. Жизненный цикл заказа (главная бизнес-логика)

```
new ─(консультант берёт в работу)→ negotiating ─(консультант прислал предложение)→ proposed
                                        ↑                                              │
                                        └──────────(клиент просит изменить)────────────┘
proposed ─(клиент согласился)→ confirmed ─(завод запустил)→ in_production ─(готово)→ completed

Отмена/отказ (терминальные): cancelled · rejected
```

### Статусы
| Статус | Что значит |
|--------|-----------|
| `new` | Клиент оформил заявку, ждёт реакции консультанта |
| `negotiating` | Консультант взял в работу, согласует с заводом (чат `factory`) |
| `proposed` | Консультант прислал клиенту итоговое предложение (цена/кол-во/срок) |
| `confirmed` | Клиент согласился с предложением |
| `in_production` | Завод запустил производство |
| `completed` | Заказ выполнен |
| `cancelled` | Отменён клиентом |
| `rejected` | Отклонён консультантом/заводом (не могут выполнить) |

### Переходы: кто и когда
| Переход | Кто инициирует | Условие |
|---------|----------------|---------|
| `new → negotiating` | консультант | «Взять в работу» |
| approve (флаг) | factory_admin | (кастом) подтверждает производимость → `factoryApproved=true` |
| `negotiating → proposed` | консультант | Заполнил предложение (qty, unitPrice, total, leadTime, note). **Для кастома** (`productId==null`) нужно `factoryApproved=true`; для каталога — без ограничения |
| `proposed → confirmed` | **клиент** | Принял предложение |
| `proposed → negotiating` | **клиент** | «Запросить изменение» (ревизия) |
| `confirmed → in_production` | **только factory_admin** | Производство запущено |
| `in_production → completed` | **только factory_admin** | Готово |
| reassign (смена консультанта) | factory_admin / platform_admin | Новый консультант — того же завода |
| → `cancelled` | клиент (`new/negotiating/proposed`) · consultant/factory_admin/admin (до `confirmed`) · после `in_production` — только platform_admin | Отмена |
| `new/negotiating → rejected` | консультант / factory_admin | Не могут выполнить |

**Правила (✅ подтверждено):**
- Согласование с заводом обязательно **только для кастома** (`productId` пустой); каталожный заказ консультант предлагает сам.
- При создании заявки (`new`) сразу создаётся чат `client`. Чат `factory` создаётся **лениво** (при первом сообщении заводу / при approve).
- Производство двигает **только `factory_admin`**. `factory_admin` **видит имя клиента** в заказе; клиент не пишет заводу напрямую.

## 6. Сценарии по ролям

### Клиент
1. Смотрит публично: список заводов, карточку завода (товары + консультанты), категории.
2. Навигация «категория → заводы → завод → консультанты».
3. Выбирает консультанта завода → оформляет заявку (товар из каталога **или** кастом-описание) + кол-во + пожелания.
4. Внутри заявки — чат с консультантом (`client`-тред).
5. Получает предложение (`proposed`) → подтверждает или просит изменить.
6. После `completed` — оставляет **два отзыва**: консультанту и заводу (см. §4 Review).
7. Никогда не видит чат консультант↔завод и не пишет заводу напрямую.

### Консультант
1. Видит каталог **своего** завода (цены, MOQ, сроки).
2. Видит входящие заявки на себя; «берёт в работу» (`new → negotiating`).
3. Чат с factory_admin (`factory`-тред): уточняет наличие/срок/цену.
4. Формирует предложение (финальная цена/кол-во/срок) → `proposed`.
5. Ведёт оба чата по заявке; двигает статусы в своей зоне.
6. Имеет публичный профиль с рейтингом и отзывами.

### factory_admin
1. Управляет профилем своего завода.
2. CRUD каталога товаров (виды, цена-диапазон, MOQ, срок, фото, категория).
3. Создаёт/удаляет консультантов своего завода.
4. Отвечает консультантам в `factory`-чатах по заказам.
5. Двигает производство: `confirmed → in_production → completed`.
6. Видит **статистику** (см. §8) по своему заводу.

### platform_admin
1. Создаёт заводы + их `factory_admin` (отдельный аккаунт).
2. Создаёт консультантов к любому заводу; создаёт клиентов.
3. Управляет категориями (CRUD).
4. Модерирует отзывы (approve/remove).
5. Видит все заводы/заказы/пользователей; может блокировать пользователей.

## 8. Статистика factory_admin (расширенная)

Дашборд завода:
- кол-во активных товаров; кол-во консультантов;
- кол-во заказов по статусам (new/negotiating/proposed/confirmed/in_production/completed/cancelled/rejected);
- завершённых сделок (`completed`); суммарная стоимость завершённых заказов (Σ `quotedTotal`);
- последние заказы (лента);
- **топ товаров** по числу заказов;
- **разбивка по консультантам** (заказов / завершено на каждого);
- **динамика по месяцам** (заказы и завершённые — для графика).

## 11. Решения (всё закрыто)

1. Пароль при создании админом — **задаётся вручную** (email + имя + пароль).
2. Отмена — по статусам (§5): client в `new/negotiating/proposed` (до `confirmed`); consultant/factory_admin/platform_admin до `confirmed`; после `in_production` — только platform_admin.
3. Отзыв — **два**: консультанту + заводу, по одному каждого типа на заказ.
4. Статистика — **расширенная** (§8).
5. Раздел «Настройки платформы» — **убран** (trial/авто-назначение не нужны).
6. Кастом-заявка: завод = завод выбранного консультанта (отдельный выбор не нужен); согласование завода для кастома обязательно.

**Прочие решения:** один товар на заявку; переназначение консультанта (того же завода); картинки — URL; валюта USD (`$`); уведомления — только в приложении; MOQ — мягко (предупреждение); удаление товара — soft (`active=false`); авто-бана по рейтингу нет; email — только сброс пароля; производство — только factory_admin.
